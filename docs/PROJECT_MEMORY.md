# 招标产品组合系统：项目记忆

更新时间：2026-07-20

本文档记录当前已经达成一致的产品文档解析、招标文档解析和候选方案生成方案。后续设计以本文档为基线；如果业务规则变化，应修改本文档并保留变更记录。

## 1. 已确定的核心原则

1. 公司拥有几百个最小子产品；投标时选择的是这些子产品组成的组合。
2. 一份产品说明书可能覆盖多个具体型号，必须拆分成型号级产品档案。
3. 具体型号是检索、需求匹配、评分和组合求解的最小产品实体。
4. 公司每个子产品只属于一个产品线/分类标签；一个顶层招标产品可以涉及多个产品线。
5. 标签/产品线只负责缩小检索范围，不能证明产品满足招标要求。
6. 产品是否满足需求，以结构化产品原子事实、兼容关系及原文证据为准。
7. 招标需求同时从两个独立维度分类：
   - 强度：硬需求、软需求。
   - 作用范围：顶层产品总需求、产品线/子产品需求、产品关系需求、全局需求。
8. 软需求用于评分、排序和优劣势分析，不能抵消硬需求不满足。
9. 不将“组合模板”作为核心必经流程。组合由产品线、产品原子事实、产品关系和约束求解器动态生成。
10. 历史已验证组合未来可以作为推荐和求解加速数据，但不能限制动态组合。
11. LLM负责文档理解、结构化抽取和方案说明；确定性规则负责参数比较、单位换算和硬需求判断；优化求解器负责组合选择。

## 2. 统一术语

| 术语                  | 定义                                               |
| ------------------- | ------------------------------------------------ |
| `ProductModel`      | 具体型号；当前业务中的最小产品实体。若同型号还有独立销售配置，再增加 `ProductSKU`。 |
| `ProductLine`       | 公司产品线/受控分类标签；每个最小子产品只属于一个产品线。                    |
| `ProductFact`       | 某个具体型号的正式原子事实，可直接参与招标判断。                         |
| `ProductRelation`   | 产品之间的依赖、兼容、互斥、包含、替代、共享等关系。                       |
| `Evidence`          | 文档原文及其文件版本、页码、表格、单元格、坐标等定位信息。                    |
| `FactCandidate`     | OCR、规则或LLM从证据中抽出的临时事实候选，不能直接用于投标判断。              |
| `TenderItem`        | 招标书中的顶层招标产品。                                     |
| `TenderRequirement` | 招标书中拆出的正式原子需求。                                   |
| `LineProductPool`   | 某顶层招标产品下，某产品线通过硬需求初筛的公司型号集合。                     |
| `ItemCandidate`     | 能够响应一个顶层招标产品的一套完整公司子产品组合。                        |
| `BidPlan`           | 从每个顶层产品的候选组合中各选一个后形成的最终投标方案。                     |

## 3. 通用文档解析层

产品资料和招标文件都可能是PDF、扫描PDF、Word、Excel、PPT或图片。各种格式先转换为统一中间结构 `DocumentIR`：

### 3.1 文档元数据

每个源文档及其版本至少保存：

- 文档名称。
- 文档类型：技术说明书、彩页、证书、检测报告或配置清单。
- 文件格式：PDF、DOCX、XLSX、PPTX或图片。
- 文件hash和文档版本。
- 发布日期、生效日期和失效日期。
- 语言。
- 页数、Sheet数或Slide数。
- 解析器版本和解析时间。

文件hash用于内容去重和完整性校验；同一文档的不同版本必须分别保存，不能用新版本覆盖历史版本。

### 3.2 统一文档结构 `DocumentIR`

```text
Document
├── Section
├── Paragraph
├── List
├── Table
│   ├── Row
│   └── Cell
├── Figure
└── Footnote
```

每个文档节点至少保存：

- 文档版本ID、节点ID、父节点ID。
- 节点类型、标题路径、阅读顺序。
- 原始文本。
- PDF页码、Excel Sheet、PPT Slide。
- 页面坐标、表格行列坐标或单元格范围。
- 图片地址。
- 适用型号范围。
- OCR/解析器版本和置信度。

示例：

```json
{
  "block_id": "B-102",
  "type": "table_cell",
  "heading_path": ["技术参数", "型号对比"],
  "page": 28,
  "row": 3,
  "column": 2,
  "text": "12导联",
  "applies_to_models": ["NL-200", "NL-300"]
}
```

原始文件和大型解析产物存入MinIO；结构化业务数据存入PostgreSQL；语义检索向量使用pgvector；RabbitMQ用于Celery任务队列；Redis只用于缓存、分布式锁和临时状态。

## 4. 产品文档解析方案

### 4.1 处理流程

```text
产品资料上传
→ 文件hash去重并保存原文件
→ 多格式解析生成DocumentIR
→ 识别产品系列和全部型号
→ 与公司产品主数据进行型号归一化
→ 提取带型号适用范围的文档证据
→ 抽取事实候选
→ 指标、数值、单位、条件和否定标准化
→ 型号范围展开
→ 去重、继承、型号专属覆盖和版本处理
→ 冲突检测及人工审核
→ 发布型号级产品原子事实
→ 建立关键词和向量索引
```

### 4.2 产品和型号清单

一份说明书可包含多个型号。必须识别：

- 产品系列、注册产品名称、商品名称和全部型号。
- 型号别名、型号之间的继承关系和型号是否停产。
- 文档公共内容适用于哪些型号。
- 章节、段落、表格行列或局部表达适用于哪些型号。
- “仅适用于”“除某型号外”“选配后支持”等限定条件。

示例：

```json
{
  "family_name": "NL系列心电设备",
  "models": [
    {
      "model_code": "NL-100",
      "aliases": ["NL100", "NL 100"]
    },
    {
      "model_code": "NL-200",
      "aliases": ["NL200", "NL 200"]
    }
  ]
}
```

每个归一化型号最终对应一个独立的 `ProductModel`。不需要物理拆分原始PDF，而是生成多个引用同一原文证据的型号虚拟档案。

### 4.3 型号适用范围

每个章节、段落、表格行或参数都必须标记适用型号。`scope_type` 使用以下标准值：

- `all_models`：适用说明书中的全部型号。
- `single_model`：仅适用一个型号。
- `model_subset`：适用部分型号。
- `family`：适用整个产品系列。
- `except_models`：除指定型号外均适用。
- `conditional`：满足特定配置或条件时适用。
- `unknown`：无法判断适用范围。

示例：

```json
{
  "scope_type": "model_subset",
  "include_models": ["NL-200", "NL-300"],
  "exclude_models": [],
  "condition": null
}
```

### 4.4 带型号适用范围的文档证据 `ScopedEvidence`

文档证据与型号适用范围合并提取为 `ScopedEvidence`。必须区分：

- 原文内容和原文位置。
- `mentioned_models`：原文明示提到的型号。
- `applicable_models`：该证据或事实实际适用的型号。
- `excluded_models`：明确排除的型号。
- `scope_source`：型号范围的判断依据。
- `scope_confidence`：型号范围判断的置信度。

`scope_source` 使用以下标准值：

- `explicit_text`：原文直接说明。
- `document_cover`：来自封面适用型号。
- `section_heading`：来自章节标题。
- `table_row_header`：来自表格行标题。
- `table_column_header`：来自表格列标题。
- `inherited_from_parent`：继承父节点范围。
- `exclusion_expression`：来自“除某型号外”等排除表达。
- `configuration_condition`：来自配置条件。
- `model_inference`：模型推断。
- `manual_review`：人工确认。
- `unknown`：无法判断。

同一证据可能包含多个事实且各自适用型号不同。因此证据可以提供默认范围，但 `FactCandidate` 仍可覆盖为更精确的事实范围。

### 4.5 产品侧需要抽取的内容

1. 产品身份：系列、具体型号、注册名称、商品名称、别名、制造商、产品线、生命周期状态。
2. 用途与场景：预期用途、科室、人群、环境、禁忌和限制。
3. 原子技术能力：数值、区间、布尔、枚举、集合和描述性能力。
4. 条件：工作模式、软件版本、选配模块、附件和环境条件。
5. 配置组成：标准配置、必选配件、可选配件、软件、耗材和数量比例。
6. 产品关系：`requires`、`compatible`、`incompatible`、`includes`、`alternative`、`connects_to`、`shares`、`quantity_ratio`。
7. 证书和证明材料：证书类型、编号、适用型号、有效期及证明能力。
8. 原文证据：文档版本、页码、表格、单元格、坐标和原文。

### 4.6 事实候选到产品原子事实

`FactCandidate` 是不可信的解析中间结果；`ProductFact` 是可以参与投标判断的正式型号级原子事实。转换过程：

```text
格式和原子性校验
→ 型号归一化
→ 指标编码归一化
→ 数值与单位标准化
→ 条件、否定和限制识别
→ 型号范围展开
→ 重复事实及证据合并
→ 公共值继承和型号专属值覆盖
→ 文档权威性、版本和有效期处理
→ 冲突检测
→ 证据质量检查
→ 自动或人工审核
→ 发布ProductFact
```

一个候选可能生成多条型号级事实；多个相同候选也可能合并成一条原子事实并关联多个证据。缺少说明不等于不支持，必须区分 `PASS`、`FAIL`、`UNKNOWN` 和 `CONFLICT`。

正式输出示例：

```json
[
  {
    "product_id": "NL-200",
    "fact_code": "ecg_lead_count",
    "value": 12,
    "unit": "lead",
    "status": "published"
  },
  {
    "product_id": "NL-300",
    "fact_code": "ecg_lead_count",
    "value": 12,
    "unit": "lead",
    "status": "published"
  }
]
```

发布后的 `ProductFact` 是招标匹配直接使用的数据；LLM抽取的候选事实不能直接参与正式匹配。

### 4.7 检索片段

产品文档还要生成用于语义检索的片段，包括：

- 产品介绍片段。
- 功能描述片段。
- 参数表行片段。
- 配置关系片段。
- 使用场景片段。
- 证书说明片段。

每个检索片段必须携带型号、产品线/分类、事实编码、文档版本、原文位置和Embedding模型版本等过滤元数据。

```json
{
  "chunk_id": "CH-901",
  "content": "NL-300支持十二导联同步采集及无线数据传输。",
  "model_ids": ["NL-300"],
  "category_id": "ECG_DEVICE",
  "fact_codes": [
    "ecg_lead_count",
    "wireless_transmission"
  ],
  "document_version_id": "DV-001",
  "page": 28,
  "embedding_model": "embedding-model-version"
}
```

技术数值和合规性最终通过结构化事实进行确定性判断；向量片段只负责召回相关内容，不能作为参数满足或合规的直接证据。

## 5. 招标文档解析方案

### 5.1 处理流程

```text
招标文件及附件上传
→ 文件版本识别并保存原文件
→ 多格式解析生成DocumentIR
→ 恢复项目、标包、顶层产品层级
→ 提取带业务作用范围的招标证据
→ 得到每个顶层产品的总需求并区分软硬
→ 识别每个顶层产品涉及的产品线
→ 得到每条产品线的子需求并区分软硬
→ 提取产品线之间的关系需求
→ 抽取并结构化评分细则
→ 提取资格、商务、材料和全局约束
→ 需求标准化、证据关联、冲突检查及审核
→ 发布正式TenderRequirement和ScoringRule
```

### 5.2 招标层级

```text
TenderProject
└── TenderLot
    └── TenderItem（顶层招标产品）
        ├── 顶层总需求
        ├── 涉及的产品线
        ├── 产品线子需求
        └── 产品关系需求
```

需要解析项目编号、招标人、标包、预算、最高限价、顶层产品名称、数量、单位、交付要求、是否允许进口或等效响应等内容。

### 5.3 需求的两个独立维度

强度：

- `hard`：不满足时淘汰候选或转人工审核。
- `soft`：用于评分、偏好、方案描述和优劣势分析。

作用范围：

- `top_product`：针对整个顶层产品组合的总需求。
- `product_line`：针对某条产品线/子产品的需求。
- `product_relation`：针对两个或多个子产品间关系的需求。
- `global`：针对多个顶层产品或整个标包的约束。

需求强度和作用范围必须分别存储，不能合并成一个枚举。

### 5.4 顶层总需求

顶层总需求描述完整组合需要具备的能力，例如整套系统需要同时完成采集、分析、打印和HIS上传。总需求可能由一个或多个公司子产品共同覆盖，不能默认拿它过滤每个单独产品。

### 5.5 产品线和子需求

每个顶层产品可以涉及多个产品线。产品线来源为：

```text
顶层产品分类结果
∪ 子需求对应的产品线
∪ 产品依赖关系引入的产品线
```

每条产品线保存其在顶层产品中的角色、是否必选、最小/最大数量、是否可共享以及该产品线的软硬子需求。一个需求可能由多条产品线替代或联合满足，因此需求与产品线是多对多关系。

### 5.6 评分规则

评分规则独立于需求保存，支持：

- 布尔得分、阶梯得分、线性得分。
- 数量得分、排名得分、价格公式。
- 主观评分和人工评分。
- 满分、阈值、公式变量、证明材料和关联需求。

同一个技术指标可以同时具有硬性最低要求和额外加分规则。价格分依赖未知竞争报价时，必须进行场景分析，不能伪造精确分数。

### 5.7 招标侧需要抽取的其他内容

- 资格要求：注册证、许可证、授权书、案例和信用条件。
- 商务要求：交货、质保、售后、培训、付款和报价口径。
- 材料要求：说明书、检测报告、彩页、盖章及证明主体。
- 全局约束：统一品牌、统一平台、共享工作站、数量比例、预算、兼容和不可拆分报价。
- 澄清和更正文件：必须版本化，不能覆盖删除原始条款。

## 6. 候选组合和最终方案流程

### 6.0 端到端标准流程

1. 解析每个顶层招标产品：提取顶层总需求，并将需求区分为软需求和硬需求。
2. 解析顶层产品包含的产品线：提取每条产品线的数量、角色和子需求，将子需求区分为软需求和硬需求，并解析产品线之间的关系需求。
3. 在每条产品线中召回公司子产品：使用产品线硬需求过滤，使用软需求、成本和证据质量排序，得到每条产品线的可用产品池 `LineProductPool`。
4. 为每个顶层产品生成子产品组合：从各产品线的可用产品池中选择产品，检查数量、依赖、兼容和互斥关系以及顶层硬需求，计算顶层和子产品软需求得分，得到每个顶层产品的 Top-K 候选组合 `ItemCandidate`。
5. 将多个顶层产品的候选组合进行全局组合：为每个顶层产品选择一个候选组合，检查预算、共享组件、统一平台等全局约束，输出若干最终投标方案 `BidPlan` 及其优劣势。

该流程中，硬需求和确定性约束用于可行性判定，软需求、成本和证据质量用于排序与多目标优化；软需求得分不能抵消任何未满足的硬需求。

### 6.1 产品线初筛

对每个顶层招标产品涉及的每条产品线：

```text
取得该产品线的全部公司型号
→ 使用产品线硬需求过滤
→ 对UNKNOWN和CONFLICT转人工审核或按风险策略处理
→ 使用软需求、成本、证据质量和风险排序
→ 得到LineProductPool
```

软需求不能作为硬淘汰条件。为避免过早丢失组合多样性，可按成本、技术分、证据和风险进行Pareto保留，而不是只保留单一加权分最高的产品。

### 6.2 顶层产品候选组合

从各产品线的可用产品池中选择一个或多个公司型号，检查：

1. 必选产品线和数量约束。
2. 产品线硬需求。
3. 产品依赖、兼容、互斥、包含和共享关系。
4. 产品关系硬需求。
5. 顶层产品硬总需求。

可行组合再计算：

- 顶层软需求得分。
- 产品线软需求得分。
- 总成本、证据完整度、组件数量和交付风险。

每个顶层产品保留若干有差异的 `ItemCandidate`，例如低成本、高技术分、少组件、高证据完整度和均衡候选。

### 6.3 最终全局方案

每个顶层产品都有自己的候选组合集。最终求解器从每个集合中各选一个，形成 `BidPlan`，同时校验：

- 标包和总预算。
- 跨顶层产品的共享组件和重复采购。
- 统一品牌、统一平台和数据兼容要求。
- 库存、交期和售后复杂度。
- 全局评分和价格策略。

最终输出多种差异化方案，并基于确定性指标生成优劣势：高技术得分、低成本、均衡和低交付风险等。

## 9. 不可违反的业务规则

1. 不允许用同一说明书中其他型号的参数替代当前型号参数。
2. 文档没有说明某项能力时，结果是 `UNKNOWN`，不是 `FAIL`，更不是 `PASS`。
3. 硬需求未知或证据冲突时不能自动判定满足。
4. 顶层硬总需求通常在组合级校验，不能一律用于过滤单个子产品。
5. 招标要求、产品事实和评分结果都必须能追溯到原文证据。
6. 正式事实和需求不能由LLM输出直接发布，必须经过规则校验、冲突检测及相应审核策略。
7. 历史文档版本和历史投标使用的事实快照不能被覆盖删除。
8. 最终方案优劣势必须来自成本、得分、证据、组件数量和风险等计算结果，不能由LLM凭空生成。

## 10. 当前暂未确定的事项

- 产品线分类树和指标字典的最终定义。
- 各类文档的权威等级和冲突解决优先级。
- `UNKNOWN` 硬需求的人工审核和放行政策。
- 价格、库存、交期和历史投标数据的来源。

## 11. 身份与权限基线

1. 系统账号角色固定为 `manager`（经理）和 `employee`（员工）。
2. 所有业务接口必须先登录；登录、刷新会话以及健康/就绪探针除外。
3. 账号只能由已登录经理创建，不提供员工自助注册。
4. 项目首次启动时，从部署密钥配置中幂等创建首个经理账号；代码、镜像和日志不得包含其明文密码。
5. 经理拥有账号创建、内容上传和内容修改权限；员工不能创建账号、上传内容或新增、编辑、删除业务数据，其他只读和执行类能力允许使用。
