## 一、项目与工程基线

- 建立根目录 `AGENTS.md`，记录架构边界、依赖方向和编码约束
- 建立 `CLAUDE.md`，只补充研发工具相关说明
- 建立 `PROJECT_MEMORY.md`，记录当前产品与业务设计基线
- 创建 `pyproject.toml`
- 配置 Python 3.13（`>=3.13,<3.14`）
- 配置 `src` 包布局
- 配置依赖分组：运行、开发、测试
- 配置 Ruff 格式化及静态检查
- 配置 Pyright 或 mypy 类型检查
- 配置 pytest、pytest-asyncio 和覆盖率
- 配置 pre-commit
- 创建 `.env.example`
- 配置 `.gitignore`
- 明确开发、测试、生产环境差异

建议最先形成：

```
backend/
├── pyproject.toml               # 也可按文档放在项目根
├── src/bid_system/
│   └── __init__.py
└── tests/
    ├── unit/
    ├── integration/
    ├── contract/
    └── e2e/
```

## 二、`bootstrap/`：应用装配与生命周期

这里负责“如何把系统组装起来”，不放业务逻辑。

```
backend/src/bid_system/bootstrap/
├── settings.py
├── dependencies.py
├── lifecycle.py
├── container.py
└── logging.py
```

清单：

- 使用 `pydantic-settings` 建立配置模型
- 加载 `.env` 和系统环境变量
- 区分 dev、test、prod
- 配置 PostgreSQL、Redis、MinIO、LLM、OCR参数
- 配置密钥引用，不在代码中保存敏感值
- 建立依赖装配入口
- 装配数据库、缓存、对象存储、外部客户端
- 使用 FastAPI `lifespan` 管理资源
- 启动时建立连接池和依赖
- 关闭时释放数据库、Redis和HTTP客户端
- 配置启动失败策略
- 避免在模块导入阶段建立网络连接

### FastAPI应用

- 决定生产环境是否关闭或保护API文档
- 配置 OpenAPI operation ID
- 增加 `/health` 存活检查
- 增加 `/readiness` 就绪检查
- 增加应用版本信息接口或响应头

### 中间件

- 请求ID中间件
- 访问日志中间件
- 请求耗时统计中间件
- CORS中间件
- Trusted Host中间件
- GZip中间件
- 安全响应头中间件
- 认证上下文中间件或依赖
- 租户上下文注入
- 请求体大小限制
- 限流机制
- 敏感接口幂等键支持
- 明确中间件注册与执行顺序

建议每次请求至少生成以下上下文：

```

user_id
tenant_id
method
path
client_ip
started_at
```

## 四、统一响应与API契约

按照文档约束：

- 对外稳定响应契约放在 `shared/contracts`
- FastAPI响应映射放在 `entrypoints/api`
- 业务模块返回DTO或Result，不直接返回FastAPI Response
- ORM对象不得作为接口响应

建议结构：

```
backend/src/bid_system/shared/contracts/
├── api.py
├── errors.py
└── pagination.py
```

清单：

- 定义统一成功响应
- 定义统一错误响应
- 定义分页响应
- 定义批量操作响应
- 定义请求ID字段
- 定义错误详情字段
- 统一时间、枚举、金额序列化
- 明确文件下载、SSE等特殊接口不强制包装

示例：

```
{
  "code": "SUCCESS",
  "message": "success",
  "data": {},
  "request_id": "01J..."
}
```

错误响应：

```
{
  "code": "VALIDATION_ERROR",
  "message": "请求参数不合法",
  "details": [],
  "request_id": "01J..."
}
```

## 五、异常体系与全局异常处理

异常需要按所属边界存放：

```
modules/<module>/domain/errors.py       # 领域异常
shared/contracts/errors.py             # 稳定错误契约
entrypoints/api/exception_handlers.py  # HTTP异常映射
```

清单：

- 定义领域异常基类
- 定义应用用例异常
- 定义资源不存在异常
- 定义业务冲突异常
- 定义状态转换异常
- 定义认证与权限异常
- 定义外部服务异常
- 定义统一错误码枚举
- 注册 Pydantic参数校验异常处理器
- 注册 FastAPI/Starlette HTTP异常处理器
- 注册领域异常处理器
- 注册数据库异常处理器
- 注册未知异常兜底处理器
- 未知异常记录完整堆栈
- 生产响应隐藏堆栈、SQL及敏感参数
- 所有错误响应携带 `request_id`
- 保留领域异常到HTTP状态码的集中映射

不要让 `domain` 层引用 FastAPI 的 `HTTPException`。

## 六、`shared/kernel/`：最小共享内核

按照文档，只放真正跨模块、无专属业务含义的内容：

```

├── ids.py
├── clock.py
├── money.py
├── units.py
├── result.py
└── pagination.py
```

清单：

- `EntityId`或统一ID类型
- UUID/ULID生成策略
- `Clock`及可替换测试时钟
- UTC时间处理规范
- `Money`及Decimal精度规则
- `UnitValue`及单位表达
- `Result`成功/失败模型
- `PageToken`或分页基础类型

准入条件：

- 至少被两个业务模块使用
- 不包含单一模块业务语义
- 有明确兼容性要求

不要创建：

```

common/service.py

```

## 十三、领域模块空壳原则

第一阶段只建立：

```
modules/
├── documents/
├── product_catalog/
├── product_knowledge/
└── reviews/
```

每个模块按实际用例逐步添加：

```
domain/
application/
infrastructure/
interfaces/
public.py
```

前置约束清单：

- `domain`不依赖FastAPI、ORM、Redis、LLM SDK
- `application`只依赖本模块domain、端口和最小shared
- `infrastructure`实现application/domain定义的端口
- `interfaces/http.py`只做协议映射
- 路由中不编写业务规则
- 其他模块只能通过`public.py`访问
- 禁止跨模块导入`infrastructure`
- 禁止为了复用ORM而共享领域实体
- 不提前创建没有运行用例的空文件

## 最终启动验收标准

进入具体业务开发前，至少应满足：

- FastAPI和Worker均可独立启动
- 配置可按环境加载并完成校验
- `/health`与`/readiness`正常
- 请求ID、访问日志和耗时记录正常
- 统一成功、错误和分页响应已确定
- 全局异常处理已覆盖验证、领域和未知异常
- 数据库Session、事务及Alembic可用
- Redis、MinIO等必要依赖可连接
- 认证与租户上下文基础接口已确定
- 单元、集成、契约、E2E测试目录和配置可运行
- API及Worker镜像可以构建
- 架构依赖方向有文档或自动化检查
- 没有`shared/utils.py`等无边界公共代码
- 没有把FastAPI、ORM或Agent依赖带入domain层
- 至少有一条“HTTP请求→事务→响应”的无业务冒烟链路通过

推荐执行顺序是：工程基线 → `bootstrap` → FastAPI/Worker入口 → 统一响应与异常 → 中间件与可观测性 → 数据库与事务 → 安全和外部适配器 → 测试/迁移/部署 → 第一个业务模块纵向链路。
