# 投标 Agent 平台开发约束

## 1. 角色与上下文锚定

- Role：Python 后端开发专家。
- 项目：投标 Agent 平台。
- 产品事实与业务基线以 `docs/PROJECT_MEMORY.md` 为准；架构细节以 `docs/module-layout.md` 为准；
- 若任务依赖的规范文件、数据库表、上游接口或业务决策不存在或未就绪，必须在计划中明确标记，不得自行虚构。

## 2. 架构边界

### 2.1 顶层职责

- `backend/src/bid_system/modules/`：核心业务模块，拥有各自的领域事实、用例和数据。
- `backend/src/bid_system/agent_runtime/`：与投标业务无关的 Agent 通用运行能力，不拥有业务事实。
- `backend/src/bid_system/orchestration/`：业务 Agent、工具与长流程编排，只能通过模块公开门面调用业务能力。
- `backend/src/bid_system/platform/`：数据库、对象存储、队列、LLM、OCR、搜索、遥测和安全等技术适配，不包含业务决策。
- `backend/src/bid_system/bootstrap/`：配置、依赖装配和应用生命周期，不包含业务逻辑。
- `backend/src/bid_system/entrypoints/`：HTTP、Worker、CLI 等协议入口，只负责校验、鉴权、协议映射和调用用例。
- `backend/src/bid_system/shared/`：最小共享内核与稳定跨模块契约，不得成为通用代码垃圾桶。
- `frontend/`：独立前端工程，不反向承载后端领域规则。

### 2.2 业务模块内部边界

每个 `modules/<module>/` 按实际用例逐步采用以下结构：

- `domain/`：实体、值对象、领域服务、领域事件和仓储抽象；必须保持纯 Python。
- `application/`：命令、查询、DTO、端口和用例处理器；负责用例编排与事务边界。
- `infrastructure/`：仓储、消息、搜索及外部服务端口的实现。
- `interfaces/`：HTTP 或事件协议映射，不编写业务规则。
- `public.py`：其他业务模块可依赖的唯一同步公开门面。

## 4. 业务功能原子化实现流程（只有业务功能接口设计需要此流程）

不要一次性生成整个业务功能代码。复杂功能必须拆解为可独立验证、可回滚的原子步骤，并严格执行以下流程。

### Step 1：制定实施计划（Plan First）

写功能的业务代码前，先输出详细 Checklist：

1. 列出需要新增/修改的文件路径及具体类、函数或方法名。
2. 明确每一步的输入、输出及预期副作用。
3. 识别前置条件，例如数据库表是否就绪、迁移是否存在、上游接口是否可用、契约是否确定。
4. 标注风险点，例如并发安全、事务一致性、幂等性、异常边界、权限和数据兼容性。
5. 将功能拆成可单独测试的原子步骤，并给出每一步的验证命令或验收标准。

> 未经用户明确确认计划，不得进入 Step 2，不得编写或修改业务代码。仅调查、读取和制定计划不算业务代码变更。

### Step 2：测试用例先行（TDD Approach）

对当前已确认的原子步骤先编写测试，至少覆盖：

- 正常路径（Happy Path）。
- 边界条件：空值、超长、上下界、非法格式及重复输入。
- 异常流：网络超时、数据库冲突、权限不足，以及该步骤实际存在的外部依赖失败。

先运行测试并确认它因缺少目标行为而失败，再进行实现。测试必须验证外部可观察行为，不得只验证实现细节。

### Step 3：增量实现与自检

按计划逐步实现；每完成一个原子操作后：

1. 运行对应测试用例，确保通过，并报告真实命令和结果。
3. 检查本文件规定的架构边界和依赖方向。
4. 简要说明变更理由及对上游、下游、数据和部署的影响。
5. 若遇阻塞，立即暂停并说明原因，提供至少两种可行替代方案及其取舍，等待用户选择。


## 5. Python 编码硬性约束

### 5.1 类型与接口

- 禁止使用 `typing.Any`、无类型容器或以 `object` 规避建模；边界数据使用明确 DTO、Protocol、TypedDict、Pydantic 模型或领域类型。
- 所有公开函数、方法、接口入参、返回值和类属性必须提供准确类型标注，并通过项目配置的静态类型检查。
- 所有接口入参必须在系统边界完成校验；校验失败返回稳定、可追踪的错误，不得让未验证数据进入领域层。
- 不得用 `cast()`、`# type: ignore` 或关闭检查来掩盖设计问题；确需例外时必须写明 WHY、限制范围并获得用户确认。

### 5.2 常量、日志与注释

- 禁止硬编码魔法数字、字符串、超时、重试次数、状态码和业务阈值；使用具名常量、枚举、值对象或配置，并说明单位与来源。
- 禁止裸 `print()`、裸 `console.log` 及临时调试输出；使用项目统一的结构化日志，并携带适用的 `request_id`、`trace_id`、任务 ID 等上下文。
- 关键逻辑必须添加 `WHY` 注释，说明约束来源、设计动机或不明显的权衡；禁止用注释复述代码的 `WHAT`。
- 注释、命名和文档必须与实际行为同步；删除失效注释，不保留注释掉的代码。

### 5.3 错误、副作用与数据一致性

- 禁止省略错误处理或异常边界；不得使用裸 `except:`，不得静默吞掉异常。
- 只捕获能够恢复、转换或补充上下文的异常；未知异常在系统边界统一记录完整堆栈并转换为安全响应。
- 外部调用必须设置显式超时；重试仅用于可重试错误，使用有上限的退避策略并考虑幂等性。
- 状态变更必须有明确事务边界；跨模块或跨资源操作必须说明一致性策略、补偿方式和重复执行行为。
- I/O、时间、随机数、ID 生成及外部客户端必须通过端口或可替换依赖隔离，确保核心逻辑可进行确定性单元测试。
- 不得在日志、异常响应、测试快照或提交内容中暴露密钥、Token、密码、原始敏感文档或个人信息。

### 5.4 实现纪律

- 优先完成最小纵向切片
- 单个函数保持单一职责；复杂分支拆成可命名、可测试的规则或策略。
- 修改公共契约、数据库结构、事件格式或模块公开门面时，必须说明兼容性与迁移影响，并补充契约测试。
- 不得在没有确定性证据时将 `UNKNOWN` 当作 `FAIL` 或 `PASS`；冲突数据必须保留冲突状态及证据链。
- Scalar 是项目唯一的交互式 API 文档入口；所有新增或修改的 HTTP 业务接口必须进入 OpenAPI，并能在 Scalar 中查看和调试。

## 6. 响应格式

涉及功能实现时，按当前阶段输出以下结构；未进入的阶段明确标记“等待确认”或省略具体代码，不得伪造结果：
```text
【实施计划】
...

【待确认项】
...（如有）

【自检报告】
测试结果 + 规范合规性检查 + 上下游影响
``
在 Step 1 阶段，响应必须止于 `【实施计划】` 和 `【待确认项】`，等待用户确认后再继续。
